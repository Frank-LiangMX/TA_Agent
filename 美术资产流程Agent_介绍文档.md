### 是什么

一个**用 AI 管理美术资产全流程**的智能平台——从资产质检、审核到入库，自动化完成，替代 TA 人工逐项检查的重复劳动。

核心技术基于业界成熟的 AI Agent 模式（ReAct 自主循环、OpenAI Function Calling、RAG 语义检索），并结合 GenericAgent 的纠正学习和 Mem0 的长期记忆机制，针对游戏 TA 领域定制化实现。

### 解决什么问题

| 现状 | 用平台后 |
|------|---------|
| 规范落地靠个人理解，项目之间千差万别 | 平台统一执行规范检查，不依赖个人 |
| 各项目资产做完就散，公司不知道自己有什么 | 统一资产数据库，跨项目可检索、可复用 |
| 开新项目时不清楚已有资源，规划靠猜 | 平台检索已有资产，分析可复用量和待开发量 |
| TA 每天花 2-3 小时手动检查资产 | 20 个资产 7 分钟自动完成 |

### 核心能力

- **AI 质检**：自动解析 FBX/贴图，AI 推断分类、材质、风格
- **智能审核**：高置信度自动通过，低置信度人工确认
- **一键入库**：重命名 + 移动 + 生成 UE5 导入脚本
- **记忆学习**：纠正一次，永远记住，准确率从 70% 升到 93%
- **语义搜索**："找一个中世纪风格的金属武器"——说人话找资产
- **数据安全**：支持本地部署大模型，项目资产数据完全不出内网

## 一、解决什么问题

### 公司级痛点：规范落地难，资产复用难

公司有技术规范，但**每个项目的执行人不同，规范在落地过程中千差万别**。A 项目的命名规则和 B 项目不一样，C 项目的面数标准和 D 项目对不上。

更关键的是：**各项目做过的资产，对公司来说都是可复用的资源，但没有统一管理。** 策划要开新项目时，不知道公司已有什么资产、能复用多少、还需要新做多少。

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **规范落地不一致** | 同一套规范，不同项目执行出来完全不同 | 项目越多越混乱，质量不可控 |
| **资产无法复用** | 每个项目从零开始，公司已有的资产没人知道在哪 | 重复开发，浪费成本 |
| **新项目规划靠猜** | 开新项目时不清楚已有资源 | 工作量和成本估算不准 |
| **质检靠人工** | TA 每天花 2-3 小时手动检查 | 效率低且标准不统一 |
| **知识随人走** | TA 离职后，项目经验和规范理解一起丢失 | 团队能力不稳定 |

### 核心解法

**一句话：统一管理公司美术资产，用 AI 自动执行规范检查，让资产可检索、可复用、可追溯。**

从 TA 的效率工具起步，逐步建设为公司的资产基础设施。

## 二、产品是什么

**美术资产智能管理平台** 是一个专为游戏美术资产全流程管理设计的 AI 智能平台。

它不是生图工具，不是编程助手，而是**理解游戏资产管线、能自动执行质检和入库的数字员工**。

### 架构

```
用户（TA / 美术）
      ↓ 自然语言交互
  ┌───────────────────────────────────────┐
  │         美术资产智能管理平台            │
  │    （LLM + TA 领域知识 + 工具集）       │
  └───┬─────┬─────┬─────┬─────┬─────┬────┘
      │     │     │     │     │     │
   资产分析 命名检查 面数检查 贴图检查 AI推断 入库执行
   (FBX解析) (规范校验) (预算控制) (分辨率/格式) (分类/材质) (重命名+移动+清单)
```

### 3.5 核心机制：项目记忆系统

记忆系统是 Agent 区别于普通脚本的核心能力——**它让 Agent 在项目中运转时间越长，分析越准。**

#### 解决的问题

传统工具每次运行都是"失忆"的：今天纠正了它的错误，明天同样的错误还会犯。Agent 的记忆系统解决了这个问题：**用户纠正一次，Agent 永远记住。**

#### 三层记忆架构

```
┌─────────────────────────────────────────────────────┐
│  L0  项目画像（Project Profile）                      │
│  "这个项目是什么样的"                                  │
│  内容：项目风格、命名约定、目录结构、美术方向            │
│  特点：全局有效，每次分析都参考，上限 500 tokens        │
├─────────────────────────────────────────────────────┤
│  L1  推断规则（Inference Rules）                       │
│  "遇到这类资产应该怎么判断"                             │
│  内容：从用户纠正中自动提炼的规律                       │
│  特点：按资产特征匹配，带置信度，最多 20 条             │
├─────────────────────────────────────────────────────┤
│  L2  纠正日志（Correction Log）                        │
│  "用户改过什么、为什么改"                               │
│  内容：每次纠正的原始记录（资产名、错误结果、正确结果）   │
│  特点：原始数据，积累到阈值后自动压缩为 L1 规则         │
└─────────────────────────────────────────────────────┘
```

#### 工作原理

```
第 1 次使用：
  Agent 分析 → "这是一栋建筑"（错误）
  用户纠正   → "这是武器，因为它是角色拿的近战道具"
  Agent 记录 → 写入 L2 纠正日志

第 2-10 次使用：
  继续积累纠正记录...

第 11 次（达到压缩阈值）：
  自动压缩 → 从 10 条纠正中提炼出规则：
    "前缀 SM_ + 面数 < 10K + 材质含 metal → 大概率是武器/道具，不是建筑"
  写入 L1 规则库，清除已压缩的日志

第 12 次以后：
  分析新资产时，Agent 自动检索相关规则注入上下文
  → 同类错误不再犯
```

#### 自动压缩机制

| 机制 | 说明 |
|------|------|
| **触发条件** | 纠正日志积累到 10 条 |
| **压缩方式** | 按"正确结果"分组，相同结论的纠正合并为一条规则 |
| **容量控制** | 规则上限 20 条，超出时保留置信度最高的 |
| **置信度计算** | `1 - (被纠正次数 / 被命中次数)`，越用越准 |
| **淘汰机制** | 置信度低的规则会被新规则挤掉，防止错误知识残留 |

#### 为什么越用越聪明

```
          纠正次数
             ↑
    ┌────────┤
    │  初期   │  频繁纠正，Agent 快速学习项目特点
    │        │
    ├────────┤
    │  中期   │  纠正减少，规则库初具规模
    │        │
    ├────────┤
    │  后期   │  偶尔纠正，微调规则置信度
    │        │
    └────────┴──→ 时间
```

- **初期**：Agent 对项目不了解，需要频繁纠正，每次纠正都在积累知识
- **中期**：规则库建立后，同类资产的分析准确率显著提升，纠正频率下降
- **后期**：Agent 已经"熟悉"这个项目，只在遇到新类型资产时才需要纠正

#### 实际效果示例

```
项目运行 1 个月后，Agent 的记忆状态：
  - 项目画像：已记录"暗黑奇幻风格，武器为主，面数预算宽松"
  - 推断规则：15 条（如"SM_ 前缀 + 有 skeleton → 可能是可装备道具"）
  - 纠正日志：47 条（已压缩为规则，剩余 7 条待压缩）

分析准确率变化：
  - 第 1 周：~70%（频繁误判建筑/道具）
  - 第 2 周：~85%（学会项目分类习惯）
  - 第 4 周：~93%（规则库成熟，偶有新类型误判）
```

### 3.6 核心机制：资产数据库与语义搜索

Agent 内置一个 SQLite 资产数据库，所有分析过的资产都会入库，支持**自然语言搜索**——用说人话的方式找到需要的资产。

#### 数据库做了什么

每次 Agent 分析一个资产，会自动生成一张"资产身份证"并存入数据库：

```
资产身份证（结构化存储）
├── 基础信息：文件名、类型、路径、入库时间
├── 几何属性：面数、顶点数、包围盒、骨骼数
├── 贴图属性：分辨率、格式、通道数、色彩空间
├── 分类信息：大类（武器/建筑/角色...）、子类、置信度
├── 视觉特征：风格（中世纪/赛博朋克...）、状态（崭新/破损...）
├── 材质结构：主材质、次材质、材质数量
└── 管理状态：审核状态、审核人、入库日期
```

#### 存储架构

```
┌─────────────────────────────────────────────────────┐
│                   SQLite 数据库                       │
│                                                      │
│  assets 表                                           │
│  ┌──────────────────────────────────────────────┐   │
│  │ asset_id │ name │ category │ style │ tri_count│   │
│  │ SM_Sword │ 剑   │ weapon   │ medieval│  5200  │   │
│  │ SM_Table │ 桌子  │ prop     │ modern │  1200  │   │
│  │ SK_Hero  │ 主角  │ character│ fantasy│ 28000  │   │
│  └──────────────────────────────────────────────┘   │
│  索引：category / subcategory / style / tri_count    │
│                                                      │
│  full_data 列：完整 JSON（身份证全字段）               │
│  结构化列：可直接 SQL 查询的关键字段                   │
└─────────────────────────────────────────────────────┘
```

- **结构化列**（category, style, tri_count 等）：建了索引，支持快速筛选
- **full_data 列**：存储完整的 JSON 身份证，需要详情时反序列化读取
- **性能**：4 万+ 资产毫秒级查询

#### 语义搜索原理

Agent 的搜索不是简单的关键词匹配，而是**自然语言理解 + 多维度评分**：

```
用户："我需要一个中世纪风格的金属武器，面数不要太高"
          ↓
    ① LLM 解析查询（自然语言 → 结构化条件）
          ↓
    {
      "category": "weapon",
      "style": "medieval",
      "materials": ["metal"],
      "max_tri_count": 10000
    }
          ↓
    ② 数据库预过滤（SQL 精确筛选缩小候选集）
          ↓
    ③ 多维度评分引擎（按权重打分）
          ↓
    ④ 排序返回 Top 结果
```

#### 评分权重

| 维度 | 权重 | 说明 |
|------|------|------|
| category（大类） | 30% | 最重要的区分维度 |
| style（风格） | 20% | 视觉风格匹配 |
| materials（材质） | 15% | 材质关键词匹配 |
| subcategory（子类） | 15% | 细分类别 |
| condition（状态） | 10% | 崭新/破损等 |
| size_class（尺寸） | 5% | 小/中/大型 |
| color_palette（色调） | 5% | 冷色/暖色等 |

每个维度独立打分，加权汇总后得出 0-100 的匹配度。

#### 实际使用示例

```
TA："帮我找几个破损风格的石头材质建筑"

Agent 调用 search_assets，返回：
  ① SM_RuinedCastle_01   匹配度 92%  命中: category, style, materials
  ② SM_BrokenWall_02     匹配度 85%  命中: category, style, materials
  ③ SM_StoneTower_01     匹配度 61%  命中: category, materials
```

#### 搜索能力一览

| 查询方式 | 示例 |
|----------|------|
| 按分类 | "找所有武器" |
| 按风格 | "赛博朋克风格的资产" |

### 3.7 核心机制：工具扩展性

Agent 不是一个封闭系统——**项目人员可以往里面加自己的工具**，让 Agent 的能力随项目需求增长。

#### 为什么需要扩展性

每个项目都有独特的需求：
- 项目 A 需要检查 UE5 材质参数是否合规
- 项目 B 需要自动给模型绑骨骼
- 项目 C 需要生成团队周报

这些需求不可能全部由 Agent 核心团队预判。**好的架构应该让使用者自己扩展能力。**

#### 扩展方式

```
现在                          未来
  │                             │
  ▼                             ▼
插件目录                      多协议接入
丢一个 .py 文件进去             HTTP / MCP
Agent 自动识别并注册            对接引擎和第三方工具
```

**方式一：插件目录（最简单）**

项目人员写一个 Python 文件，放到指定目录，Agent 自动识别：

```
tools/plugins/
├── ue5_material_check.py    ← 检查 UE5 材质参数
├── blender_auto_rig.py      ← 自动绑骨
└── team_report.py           ← 团队周报
```

插件格式统一：声明工具名称、参数、描述，实现一个函数。**和核心工具完全同等对待，LLM 分不出区别。**

**方式二：远程工具（接引擎时）**

当工具运行在其他进程或机器上时（如 UE5 Editor 内），通过 HTTP 调用：

```
Agent ←→ HTTP ←→ UE5 Editor 内的工具服务
```

**方式三：MCP 协议（平台化时）**

MCP 是 AI Agent 领域的开放标准（Anthropic 定义），专门解决"外部工具接入"问题：

```
Agent ←→ MCP ←→ UE5 工具服务
Agent ←→ MCP ←→ Maya 工具服务
Agent ←→ MCP ←→ 自研引擎工具服务
```

协议统一，各引擎各自实现自己的工具服务，Agent 自动发现并调用。

#### 三种方式共存

不管用哪种方式接入，工具最终都注册到同一个地方。对 LLM 来说：

```
"我有个工具叫 check_material，参数是 material_path"
```

至于这个工具是本地 Python 函数、远程 HTTP 接口、还是 MCP Server 上的——LLM 不关心，也不需要关心。

#### 扩展路线

| 阶段 | 方式 | 适合场景 | 时机 |
|------|------|---------|------|
| 1 | 插件目录 | 团队自定义 Python 工具 | **现在** |
| 2 | HTTP 远程工具 | 对接 UE5/Unity 引擎 | 接引擎时 |
| 3 | MCP 协议 | 多引擎/多 DCC 统一接入 | 平台化时 |

#### 平台化愿景

```
┌─────────────────────────────────────────────┐
│              美术资产智能管理平台                    │
│         （统一的工具调度 + LLM 推理）          │
├──────┬──────┬──────┬──────┬──────┬──────────┤
│核心工具│项目插件│ UE5  │Unity │ Maya │ 自研引擎  │
│(内置)  │(目录) │(HTTP)│(HTTP)│(MCP) │ (MCP)    │
└──────┴──────┴──────┴──────┴──────┴──────────┘
```

Agent 从"一个工具"进化为"一个平台"——核心能力由团队维护，项目能力由项目人员扩展，引擎能力由各引擎团队通过标准协议接入。

## 3.8 里程碑规划：从工具到平台的四步路线

从单个 TA 的效率工具到公司的资产和成本管理平台，分四步走。每一步都有独立的交付价值：

```
里程碑 1 ──→ 里程碑 2 ──→ 里程碑 3 ──→ 里程碑 4
本地资产检查    资产自动入库    项目级管理平台    公司级资产助手
 （已完成）      （已完成）       （下一步）        （远期目标）
```

## 五、技术亮点与能力来源

### 5.1 业界 Agent 能力借鉴

平台的核心机制并非凭空设计，而是借鉴了 AI Agent 领域的成熟模式，并结合游戏 TA 领域做了定制化：

| 平台能力 | 借鉴来源 | 说明 |
|----------|---------|------|
| **记忆系统（L0/L1/L2）** | GenericAgent + Mem0 | GenericAgent（arXiv:2604.17091，A3 Lab + 复旦大学）的技能结晶和"No Execution, No Memory"原则；Mem0 的按需检索和自动压缩。我们在此基础上设计了三层记忆架构。GenericAgent 在 Lifelong AgentBench 测试中实现 100% 任务完成率，Token 消耗仅为 Claude Code 的 27.7% |
| **工具调用（Function Calling）** | OpenAI Function Calling | LLM 通过 JSON Schema 描述工具，自动决定调用哪个工具、传什么参数。我们用原生 API 实现，不依赖 LangChain 框架 |
| **自主工作流（ReAct 循环）** | ReAct 模式 | Reasoning + Acting 循环：LLM 思考→调用工具→观察结果→继续思考。我们的 agent_loop 就是标准 ReAct 实现，最多 10 轮迭代 |
| **语义搜索** | RAG（检索增强生成） | 自然语言查询→LLM 解析为结构化条件→数据库检索→评分排序。参考了 LangChain/LlamaIndex 的 RAG 模式 |
| **自动压缩与淘汰** | Mem0 的记忆管理 | 纠正日志积累到阈值自动提炼为规则，低置信度规则被新规则挤掉。参考 Mem0 的 ADD/UPDATE/DELETE/NOOP 决策机制 |
| **MCP 协议扩展** | Anthropic MCP | AI Agent 工具调用的开放标准，支持自动发现和调用外部工具。我们规划作为平台化的标准接入协议 |
| **分级审核（置信度）** | 我们的创新 | 业界 Agent 通常"全信"或"全不信"，我们设计了置信度阈值机制：高置信度自动通过，低置信度人工确认，在效率和准确性之间取平衡 |
| **资产身份证系统** | 我们的创新 | 业界没有"给每个资产建结构化身份证"的先例。传统工具管文件名和路径，我们管资产的完整身份（物理+材质+视觉+关联） |
| **项目记忆自动学习** | 我们的创新 | 结合 GenericAgent 的纠正学习和 Mem0 的长期记忆，但加入了 TA 领域特征（前缀、面数、材质）作为记忆匹配维度 |

### 5.2 数据安全：支持全链路本地部署

游戏项目的美术资产（角色原画、场景模型、材质贴图）是核心商业机密。多数 AI 工具要求将数据上传到云端 API，存在泄露风险。**我们的平台从架构设计上就支持完全本地化部署。**

#### 两种部署模式

```
模式一：云端 API（开发阶段，快速验证）
┌──────────┐     HTTPS      ┌──────────────┐
│   平台    │ ──────────────→ │ 云端 LLM API │
│           │                 │ GLM/DeepSeek │
└──────────┘                 └──────────────┘
  资产数据经过加密传输到云端 LLM 处理

模式二：本地部署（生产阶段，数据不出内网）
┌──────────┐     内网       ┌──────────────┐
│   平台    │ ──────────────→ │ 本地算力集群  │
│           │                 │ 部署开源 LLM │
└──────────┘                 └──────────────┘
  所有数据在内网闭环，零外泄风险
```

#### 为什么能做到

| 设计决策 | 如何支撑本地化 |
|---------|--------------|
| **LLM 抽象层** | 平台通过统一接口调用 LLM，切换模型只改配置（base_url + api_key），不改业务代码 |
| **OpenAI 兼容格式** | 国产开源模型（Qwen、GLM、DeepSeek）均兼容 OpenAI API 格式，本地部署后直接对接 |
| **零框架依赖** | 不依赖 LangChain 等重框架，减少外部依赖带来的数据流转环节 |
| **本地数据库** | TagStore 使用 SQLite，所有资产标签数据存储在本地，不依赖云服务 |

#### 本地部署方案

| 组件 | 云端方案 | 本地方案 |
|------|---------|---------|
| LLM 推理 | GLM-5 / DeepSeek API | 本地 GPU 集群 + vLLM/TGI 部署开源模型 |
| 标签数据库 | SQLite（本地） | SQLite（本地） |
| 向量知识库 | 可选云端 | 本地 ChromaDB / FAISS |
| 文件存储 | 本地 | 本地 |

**推荐的开源模型选型**（均可本地部署）：

| 模型 | 参数量 | 优势 | 最低 GPU 需求 |
|------|--------|------|-------------|
| Qwen2.5-72B | 72B | 综合能力强，中文理解好 | 2×A100 80G |
| DeepSeek-V3 | 671B (MoE) | 推理能力最强 | 4×A100 80G |
| Qwen2.5-14B | 14B | 轻量级，成本低 | 1×A100 40G |
| GLM-4-9B | 9B | 最轻量，适合初期验证 | 1×A10 24G |

#### 安全对比

| | 云端 API | 本地方案 |
|---|---|---|
| 资产数据 | 加密传输到云端处理 | **完全不出内网** |
| 模型输出 | 经过云端返回 | **内网闭环** |
| 合规风险 | 需评估数据出境/第三方风险 | **零外部依赖** |
| 成本 | 按 token 付费 | **一次性硬件投入** |
| 延迟 | 取决于网络 | **内网低延迟** |

**结论**：开发阶段用云端 API 快速验证，生产部署切换到本地算力，**一套代码，两种部署，零改动**。

### 5.3 技术架构亮点

| 特点 | 说明 |
|------|------|
| **零框架依赖** | 直接调用 LLM API（OpenAI 兼容格式），不依赖 LangChain 等框架，可控性强 |
| **LLM 可切换** | 当前使用 GLM-5 / DeepSeek，切换到 Claude/GPT 只改配置，不改代码 |
| **项目隔离** | 每个项目独立配置和记忆，互不干扰 |
| **渐进式自动化** | 支持"逐步确认"和"全自动"两种模式，适应不同熟练度的用户 |
| **UE5 原生集成** | 生成的导入脚本直接使用 unreal Python 模块，无需额外插件 |

## 七、竞品分析

### 市面上有什么

当前游戏开发领域有几类工具与我们的平台有部分重叠，但**没有任何一个覆盖了完整的"质检→审核→入库"链路**。

### 7.1 生产管理类

#### ShotGrid / Flow Production Tracking（Autodesk）

影视和游戏行业最广泛使用的生产管理工具，腾讯、网易、米哈游、暴雪、EA 等均在使用。

| 它能做 | 它不能做 |
|--------|---------|
| 追踪资产制作进度、版本管理 | 检查资产质量（面数、命名、贴图） |
| 团队排期、资源分配 | 理解资产内容（FBX/贴图内部数据） |
| AI 智能排程（2025 新功能） | 自动重命名、移动、入库 |
| DCC 工具内嵌检阅 | 语义级资产分类推断 |

**核心差异**：ShotGrid 管的是"人和进度"，我们管的是"资产质量"。它告诉你"这个资产谁在做、做到哪了"，我们告诉你"这个资产合不合规、哪里有问题"。**两者互补，不冲突。**

#### CGTeamWork

国内团队用得较多的生产管理工具，支持 Docker 私有部署。

| 它能做 | 它不能做 |
|--------|---------|
| 线性流程管理 | AI 推断和语义分析 |
| 资源复用分析 | 自动质检 |
| 私有化部署 | 理解资产内容 |

**核心差异**：偏流程管理，不涉及资产内容分析。

### 7.2 AI 生成类

#### Meshy / Tripo / Rodin

AI 3D 模型生成工具，输入文字或图片，输出 3D 模型。

| 它能做 | 它不能做 |
|--------|---------|
| 文字/图片生成 3D 模型 | 检查已有资产是否合规 |
| AI 生成贴图 | 管理入库流程 |
| 快速原型制作 | 理解项目规范 |

**核心差异**：它们解决的是"资产从哪来"，我们解决的是"资产怎么管"。**生成和管理是两个完全不同的环节。**

### 7.3 引擎内置工具

#### UE5 Asset Audit / Validation

UE5 自带的资产验证机制，支持在 Editor 内配置规则检查。

| 它能做 | 它不能做 |
|--------|---------|
| 基于规则的资产验证 | AI 推断（分类、材质、风格） |
| 集成到 CI/CD 流水线 | 自然语言交互 |
| 命名规范检查 | 跨引擎支持（仅 UE5） |
| 自动化报告 | 审核工作流、记忆学习 |

**核心差异**：纯规则驱动，没有 AI 能力。你得自己写死每条规则，不能像平台一样"理解"资产。而且只在 UE5 内可用，换引擎就失效。

#### Unity Muse

Unity 官方 AI 工具，目前以 AI 对话和代码生成为主。

| 它能做 | 它不能做 |
|--------|---------|
| AI 对话辅助开发 | 资产质量检查 |
| 代码生成 | 入库流程管理 |
| 自然语言操作编辑器 | 资产身份证/标签系统 |

**核心差异**：Unity Muse 是通用 AI 助手，不专注于 TA 流程。且项目已迁移至 UE5，Unity 生态不再适用。

### 7.4 自研脚本方案

大多数 TA 团队的现状：自己写 Python 脚本做检查。

| 它能做 | 它不能做 |
|--------|---------|
| 定制化检查规则 | AI 推断（全靠手写规则） |
| 集成到现有流程 | 自然语言交互 |
| 零成本启动 | 跨项目复用 |
| | 知识沉淀和记忆 |
| | 审核工作流 |

**核心差异**：自研脚本是"工具"，我们的平台是"数字员工"。工具需要人操作，平台能自主决策。而且每个项目的脚本不通用，平台通过项目配置实现一套代码多项目复用。

### 7.5 对比总结

|------|----------|---------|------------|---------|---------------------|

| 审核工作流 | 部分 | ❌ | ❌ | ❌ | **✅ 分级审核** |
| 入库自动化 | ❌ | ❌ | ❌ | 部分 | **✅ 全流程** |
| 自然语言交互 | ❌ | ❌ | 部分 | ❌ | **✅** |
| 记忆/学习 | ❌ | ❌ | ❌ | ❌ | **✅ 越用越准** |
| 跨引擎支持 | 部分 | 仅 UE5 | 仅导出 | 需重写 | **✅ 可扩展** |
| 数据安全 | 云端 | 云端 | 云端 | 本地 | **✅ 支持本地部署** |

### 7.7 我们的核心优势

1. **全流程覆盖**：从分析到入库一条链路，不需要拼凑多个工具
2. **AI 驱动**：不是纯规则检查，能"理解"资产（分类、材质、风格、状态）
3. **会成长**：记忆系统让平台越用越准，自研脚本做不到
4. **说人话**：自然语言交互，不需要记命令、写脚本
5. **可扩展**：插件目录 + HTTP + MCP，项目人员可以往里加自己的工具
6. **引擎无关**：核心逻辑不绑定 UE5/Unity，通过配置和插件适配不同引擎
7. **数据安全**：支持本地部署大模型，项目资产数据完全不出内网，满足保密要求

### 7.8 垂直领域 vs 通用技能平台

当前市面上的 Agent 平台（Coze、AgentGPT、Manus 等）主打"技能市场"——用户可以给 Agent 装各种插件，覆盖通用场景。

**我们的定位完全不同：不是通用技能平台，而是游戏美术领域的垂直 Agent。**

| 维度 | 技能型 Agent（通用平台） | 本平台（垂直领域） |
|------|----------------------|----------------|
| 覆盖范围 | 广泛，多行业多场景 | 窄而深，只做游戏美术资产 |
| 核心价值 | 技能数量多 | 领域知识深 |
| 工具来源 | 用户自己装插件 | 内置 30 个领域专用工具 |
| 记忆能力 | 通用键值对存储 | 针对资产特征的三层记忆架构 |
| 数据资产 | 无 | 结构化资产数据库，越用越有价值 |

**行业趋势**：2025 年行业报告指出，"99% 的通用 Agent 产品在泛用方向上很快会死亡"，而垂直领域 Agent 因为有深度领域知识和工作流整合能力，B 端落地更成熟。资本也在从通用平台转向垂直领域。

**我们的壁垒**：领域 Know-How（理解 FBX 结构、贴图通道、面数预算）+ 数据积累（结构化资产数据库）+ 记忆系统（针对资产特征的精准匹配）。这些不是装个插件就能获得的。

## 八、从工具到平台：团队协作架构

当前平台是 TA 个人使用的 CLI 工具。要成为团队基础设施，需要解决三个问题：**怎么看、谁能用、怎么分组**。

### 8.1 Web 项目看板

#### 需求

项目负责人需要随时看到全局状态，而不是打开终端跑命令：

- 当前有多少资产在审核中 / 已通过 / 已入库
- 各组进度如何
- 哪些资产有问题待解决

#### 方案

```
┌──────────┐      HTTP       ┌──────────────┐      SQLite      ┌──────────┐
│ 浏览器    │ ←──────────────→ │  Web Server  │ ←──────────────→ │ TagStore │
│ (看板)    │                 │  (FastAPI)   │                  │ (tags.db)│
└──────────┘                 └──────────────┘                  └──────────┘
                                    ↑
                                    │ 读同一个数据库
                                    ↓
                             ┌──────────────┐
                             │    Agent      │
                             │ (CLI/后台)    │
                             └──────────────┘
```

**关键设计**：Web 服务和平台共享同一个 TagStore 数据库。**平台不需要在线，Web 端也能查看数据。**

**技术选型**：FastAPI（Python，和技术栈一致），前端用轻量级方案（HTML/JS 或 Vue）。

#### 看板功能

| 模块 | 内容 | 价值 |
|------|------|------|
| 资产总览 | 按状态分组（待审核 / 已通过 / 已入库），数量统计 | 一眼看到全局 |
| 质检报告 | 按问题类型统计（命名不合规、面数超标等） | 发现共性问题 |
| 时间线 | 最近 7 天的入库记录、审核记录 | 追踪进度 |
| 团队视图 | 按组查看各自的资产状态 | 组间对比 |

### 8.2 权限管理

#### 需求

不同角色需要不同权限，跨组别的内容管理必须有权限控制：

| 角色 | 能看什么 | 能做什么 |
|------|---------|---------|
| 美术 | 只看自己的资产 | 提交资产、查看审核结果 |
| 组内 TA | 看本组所有资产 | 审核、入库 |
| 项目 TA Lead | 看所有组 | 审核、入库、改配置 |
| 项目经理 | 看所有组（只读） | 看报表，不操作 |
| 管理员 | 全部 | 全部 |

#### 方案

**轻量级方案（初期）**：在 TagStore 内建用户和团队表，平台操作时检查权限。

```sql
-- 用户表
CREATE TABLE users (
    user_id   TEXT PRIMARY KEY,
    username  TEXT,
    role      TEXT,    -- artist / ta / lead / pm / admin
    team_id   TEXT
);

-- 资产关联团队
ALTER TABLE assets ADD COLUMN team_id TEXT;
```

**标准方案（后期）**：接入企业现有账号体系（LDAP / 企业微信 / 飞书），不在平台内重复造轮子。

**建议时机**：Web 端上线后再加权限，CLI 阶段默认全权限。

### 8.3 分组管理

#### 需求

不同美术组的工作内容和检查标准完全不同：

| 组别 | 关注点 | 命名前缀 | 面数预算 |
|------|--------|---------|---------|
| 角色组 | 骨骼、面数、动画、渲染预览 | SK_ | < 30K |
| 场景组 | LOD、光照、碰撞、性能 | SM_, B_, V_ | < 20K |
| 特效组 | 粒子数、材质复杂度 | FX_, PS_ | 粒子 < 5K |
| UI 组 | 分辨率、格式、适配 | T_UI_ | - |

每个组需要独立的检查规则、独立的工作流，但共享基础设施（TagStore、入库流程、记忆系统）。

#### 方案

扩展项目配置，支持多团队定义：

```yaml
# .ta_agent/configs/project/my_game.yaml
project_name: "My Game"
engine: UE5

teams:
  character:
    name: "角色组"
    prefixes: ["SK_"]
    face_budget: 30000
    check_rules: ["skeleton", "uv", "render_preview"]
    intake_dir: "/Game/Characters/"

  environment:
    name: "场景组"
    prefixes: ["SM_", "B_", "V_"]
    face_budget: 20000
    check_rules: ["lod", "collision", "lightmap"]
    intake_dir: "/Game/Environment/"

  vfx:
    name: "特效组"
    prefixes: ["FX_", "PS_"]
    particle_limit: 5000
    check_rules: ["material_complexity"]
    intake_dir: "/Game/Effects/"
```

平台根据资产前缀自动匹配到对应组，使用该组的规则检查。

#### 架构

```
┌─────────────────────────────────────────────────┐
│                  平台核心                              │
│     （共享：TagStore、入库流程、记忆系统、LLM）      │
├─────────────────────────────────────────────────┤
│                  团队配置层                        │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ 角色组    │  │ 场景组    │  │ 特效组    │       │
│  │          │  │          │  │          │       │
│  │ 前缀:SK_ │  │ 前缀:SM_ │  │ 前缀:FX_ │       │
│  │ 面数:<30K│  │ 面数:<20K│  │ 粒子:<5K │       │
│  │ 需要渲染 │  │ 需要LOD  │  │ 需要材质  │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└─────────────────────────────────────────────────┘
```

### 8.4 实现路线

三个功能的依赖关系：

```
                 ┌──────────┐
                 │ 分组管理  │  ← 最先做，影响数据模型
                 └────┬─────┘
                      │
                 ┌────▼─────┐
                 │ Web 看板  │  ← 第二步，需要分组数据才有意义
                 └────┬─────┘
                      │
                 ┌────▼─────┐
                 │ 权限管理  │  ← 最后做，依赖 Web 端和用户系统
                 └──────────┘
```

| 功能 | 优先级 | 依赖 | 建议时机 |
|------|--------|------|---------|
| 分组管理 | 高 | ProjectConfig 扩展 | TagStore 稳定后立即做 |
| Web 看板 | 中 | FastAPI + 分组数据 | CLI 流程跑通后 |
| 权限管理 | 低 | Web 端 + 用户系统 | 团队实际使用后再加 |

**分组管理影响数据模型**（TagStore 加 team_id 字段），应尽早做，否则后面数据迁移成本高。Web 和权限可以后续叠加，不影响核心流程。
