# 美术资产智能管理平台

> liangmingxuan | 2026.05

---

## 一句话

**统一管理公司美术资产，用 AI 自动执行规范检查，让资产可检索、可复用、可追溯。**

---

## 解决什么问题

### 公司级痛点：规范落地难，资产复用难

公司有技术规范，但**每个项目的执行人不同，规范在落地过程中千差万别**。A 项目的命名规则和 B 项目不一样，C 项目的面数标准和 D 项目对不上。规范写在文档里，实际执行靠各自理解。

更关键的是：**各项目做过的资产，对公司来说都是可复用的资源，但没有统一管理。** 策划要开新项目时，不知道公司已有什么资产、能复用多少、还需要新做多少，全凭经验和感觉。

| 痛点 | 影响 |
|------|------|
| **规范落地不一致** | 同一套规范，不同项目执行出来完全不同，项目越多越混乱 |
| **资产无法复用** | 每个项目从零开始，公司已有的资产没人知道在哪、能不能用 |
| **新项目规划靠猜** | 开新项目时不清楚已有资源，工作量和成本估算不准 |
| **质检靠人工** | TA 每天花 2-3 小时做重复检查，效率低且标准不统一 |
| **知识随人走** | TA 离职后，项目经验和规范理解一起丢失 |

### 平台如何解决

| 痛点 | 平台方案 |
|------|---------|
| 规范落地不一致 | 平台内置项目配置，规范检查由 AI 统一执行，不依赖个人理解 |
| 资产无法复用 | 所有资产入库时自动生成结构化身份证，存入统一数据库，支持跨项目搜索 |
| 新项目规划靠猜 | 对平台说"我要做一个奇幻 RPG"，平台检索已有资产，分析可复用量、待开发量、成本预估 |
| 质检靠人工 | AI 自动质检，标准统一，20 个资产 7 分钟完成 |
| 知识随人走 | 记忆系统沉淀项目经验，不随人员流动丢失 |

### 从"项目工具"到"公司资产大脑"

```
现状：各项目各自为战

│ 项目 A    │  │ 项目 B    │  │ 项目 C    │
│ 自己的规范 │  │ 自己的规范 │  │ 自己的规范 │

有了平台后：统一管理，资产可复用
┌─────────────────────────────────────────┐
│           统一资产数据库                   │

开新项目时：
  策划："我要做一个暗黑风格的 ARPG"
  平台检索 → 发现项目 A 有 200 个暗黑风格资产可复用
```

**这不是一个 TA 的效率工具，是公司的资产基础设施。**

> **说明**：上述愿景需要逐步实现。当前平台已完成"质检→审核→入库"的核心链路，资产数据库和检索能力已具备基础。跨项目资产复用、新项目规划分析等功能是后续迭代方向，同时也在等待 AI 领域其他技术（如 AI 生成模型的质量和稳定性）的持续发展。我们选择**从确定性最高的质检入库切入，逐步扩展到资产管理全链路**。

## 里程碑规划

从单个 TA 的效率工具到公司的资产和成本管理平台，分四步走。每一步都有独立的交付价值，不依赖后续步骤也能用：

```
里程碑 1 ──→ 里程碑 2 ──→ 里程碑 3 ──→ 里程碑 4
本地资产检查    资产自动入库    项目级管理平台    公司级资产助手
 （已完成）      （已完成）       （下一步）        （远期目标）
```

### 里程碑 1：本地资产检查 ✅

**做了什么**：AI 自动检查单个目录下的资产（面数、贴图、命名、材质）。

**交付价值**：TA 不再逐个手动检查，20 个资产从 2 小时缩短到 1 分钟。

**状态**：已完成。

### 里程碑 2：资产自动入库 ✅

**做了什么**：审核通过后自动重命名、移动文件、生成引擎导入配置和脚本。

**交付价值**：入库流程从 7 个工具手动切换变为一键完成。

**状态**：已完成。

### 里程碑 3：项目级多维度管理平台

**做什么**：
- 所有资产入库后自动存入数据库，带结构化身份证
- 支持按分类、风格、材质、面数等多维度检索
- Web 看板：项目负责人随时查看资产状态和进度
- 多团队分组管理：不同组独立规则和工作流

**交付价值**：
- 项目内资产可检索、可追溯，不再"做完就散"
- 项目负责人有全局视图，不再靠问人了解进度

**前置依赖**：里程碑 1、2 已完成（资产需要先入库，才有数据可管理）。

### 里程碑 4：公司级资产和成本预算助手

**做什么**：
- 跨项目资产统一数据库
- 新项目规划时，检索公司已有资产，分析可复用量和待开发量
- 基于资产数据估算工作量、成本、人员需求
- 接入 AI 生产工具（模型/贴图生成），串联生产链

**交付价值**：
- 开新项目不再从零开始，公司已有资产可直接复用
- 策划提出想法，平台输出可行方案和资源清单
- 成本估算有数据支撑，不再靠经验猜

**前置依赖**：
- 里程碑 3 已完成（需要项目级数据积累到一定规模）
- AI 领域技术持续发展（生成模型的质量和稳定性）

### 每一步的价值递进

| 里程碑 | 解决谁的问题 | 解决什么问题 | 可量化指标 |
|--------|------------|------------|-----------|
| 1. 本地检查 | TA 个人 | 质检效率低 | 单次检查时间从 2 小时→1 分钟 |
| 2. 自动入库 | TA 个人 | 入库流程碎片化 | 入库步骤从 10+ 步→1 步 |
| 3. 项目管理 | 项目团队 | 资产不可查、进度不可见 | 资产检索从"翻文件夹"→秒级查询 |
| 4. 公司助手 | 管理层/策划 | 资产不复用、规划靠猜 | 新项目资产复用率、成本预估准确度 |

**饭要一口一口吃，但每一口都有营养。**

## 平台做了什么

### 核心流程：AI 自主调度

平台的核心不是"执行预设脚本"，而是 **AI 自己思考、自己决定调用什么工具、自己判断结果**：

```
TA 说一句话："帮我分析这个目录并入库"
        ↓
┌─────────────────────────────────────────────────────┐
│  AI 自主决策循环（ReAct 模式）                         │
│                                                      │
│  思考："需要先检查有没有项目配置"                        │
│    → 调用：检查项目配置工具                              │
│    → 观察："没有配置，需要询问用户"                       │
│                                                      │
│  思考："用户同意创建，创建后加载配置"                     │
│    → 调用：创建配置 → 加载配置                           │
│    → 观察："配置已加载，开始扫描目录"                     │
│                                                      │
│  思考："发现 15 个 FBX + 42 张贴图，逐个分析"            │
│    → 调用：FBX 解析 → 贴图检查 → 命名校验 → AI 推断     │
│    → 观察："12 个高置信度，3 个低置信度"                  │
│                                                      │
│  思考："高置信度批量通过，低置信度展示给用户确认"           │
│    → 调用：批量审核工具                                  │
│    → 观察："用户确认全部通过"                            │
│                                                      │
│  思考："全部通过，执行入库"                               │
│    → 调用：批量入库工具                                  │
│    → 观察："入库完成，生成导入清单"                       │
│                                                      │
│  输出："入库完成，请在 UE5 中运行导入脚本"                │
└─────────────────────────────────────────────────────┘
```

**关键点**：每一步调用什么工具、参数是什么、下一步做什么，都是 AI 实时决策的，不是写死的流程。遇到异常（文件损坏、API 超时）也能自主处理。

### 我们 vs 技能型 Agent：为什么做垂直而非通用

当前市面上的 Agent 平台（Coze、AgentGPT、Manus 等）主打"技能市场"——用户可以给 Agent 装各种插件/技能，覆盖写邮件、做 PPT、查天气等通用场景。

**我们的定位完全不同：不是通用技能平台，而是游戏美术领域的垂直 Agent。**

| 维度 | 技能型 Agent（通用平台） | 本平台（垂直领域） |
|------|----------------------|----------------|
| **覆盖范围** | 广泛，多行业多场景 | 窄而深，只做游戏美术资产 |
| **核心价值** | 技能数量多 | 领域知识深 |
| **工具来源** | 用户自己装插件 | 内置 20+ 领域专用工具 |
| **记忆能力** | 通用键值对存储 | 针对资产特征的三层记忆架构 |
| **数据资产** | 无 | 结构化资产数据库，越用越有价值 |
| **落地效果** | 通用但不深入 | 单点突破，20 倍效率提升 |

**行业趋势判断**：2025 年行业报告指出，"99% 的通用 Agent 产品在泛用聊天助手方向上很快就会死亡"，而垂直领域 Agent 因为有深度领域知识和工作流整合能力，B 端落地更成熟、价值更明确。资本也在从通用平台转向垂直领域。

**我们的核心壁垒**：
- **领域 Know-How**：理解 FBX 结构、贴图通道、命名规范、面数预算——这些不是装个插件就能获得的
- **数据积累**：每个项目的资产都沉淀为结构化数据，形成公司级资产数据库
- **记忆系统**：针对资产特征（前缀、面数、材质）的记忆匹配，比通用记忆更精准

### 技术基础：借鉴 GenericAgent 的研究成果

平台的记忆和学习机制参考了 A3 Lab（深圳 Aquaintelling + 复旦大学联合团队）的 **GenericAgent** 项目（arXiv:2604.17091）。该研究已发表论文并通过了量化验证：

**GenericAgent 核心数据**（Lifelong AgentBench 基准测试）：

| 指标 | 数据 |
|------|------|
| 任务完成率 | 100% |
| Token 消耗 vs Claude Code | 仅 27.7% |
| Token 消耗 vs OpenClaw | 仅 15.5% |
| 重复任务 9 轮后 Token 下降 | 89.6% |
| LLM 调用次数收敛 | 32 次 → 5 次 |

**借鉴的核心思想**：
- **技能结晶**：完成任务后自动将执行路径固化为可复用技能（我们的记忆系统 L1 层规则就是这个思想）
- **No Execution, No Memory**：只有经过验证的结果才写入记忆，防止幻觉污染
- **分层记忆 + 按需加载**：深层记忆不全量加载，按相关性检索后注入上下文

**我们的差异化**：GenericAgent 是通用的计算机操作 Agent，我们把它"技能结晶"和"分层记忆"的思想应用到了游戏美术这个特定领域，并加入了置信度分级审核、资产身份证等领域的创新。

平台的核心能力建立在业界成熟的 AI 技术之上：

## 长期方向：接入美术生产链的 AI 工具

### 行业趋势

各公司正在美术生产的每个环节引入 AI 工具：

| 生产环节 | AI 工具（已有或研发中） | 成熟度 |
|---------|----------------------|--------|
| **模型生成** | 混元 3D、Tripo3D、Meshy | 已商用，但生成质量仍需人工调整 |
| **贴图生成** | GPT-Image、ComfyUI | 已商用，风格可控性在提升 |
| **绑骨动画** | AI 绑骨、动捕工具 | 部分可用，精度仍依赖人工修正 |
| **LOD 生成** | Simplygon、自动减面 | 已成熟 |

**现状**：这些工具各自独立，美术需要在多个工具之间手动切换，数据传递靠人工。每个工具的输入输出格式不同，串接成本高。

### 我们的切入点

平台已经具备 AI 自主调度工具的能力（当前 20+ 工具），接入新工具只需实现一个插件。但接入 AI 生产工具不是简单"调一下 API"，需要解决几个实际问题：

| 问题 | 说明 | 难度 |
|------|------|------|
| **输入输出标准化** | 每个 AI 工具的 API 格式不同，需要写适配层 | 中，工程量大但技术明确 |
| **质量兜底** | AI 生成的模型/贴图质量不稳定，需要质检环节过滤 | 低，我们已有质检能力 |
| **人工介入点** | 哪些环节可以全自动，哪些必须人工确认 | 中，需要实际测试才知道 |
| **工具成熟度依赖** | 如果 AI 工具本身不可靠，串联起来也没意义 | 高，取决于外部工具 |

### 可行的路径

不是一步到位做成"全自动生产线"，而是分步接入：

**第一步（可做）**：接入 AI 生成工具作为"素材来源"
```
美术用混元 3D 生成模型 → 手动导出 → 喂给平台质检 → 入库
```
平台负责"兜底"——不管模型从哪来，都要过质检和入库。这一步不需要改变现有工作流。

**第二步（技术可行，需验证）**：平台直接调用 AI 生成工具
```
美术在平台内指定"生成一个中世纪风格的剑" → 平台调用混元 3D API → 自动质检 → 入库
```
技术上可行（调 API），但依赖 AI 生成工具的 API 开放程度和生成质量。

**第三步（远期探索）**：多工具串联
```
平台调用模型生成 + 贴图生成 + 绑骨工具 → 自动组合 → 质检 → 入库
```
需要各工具 API 都成熟，且标准化输入输出格式。目前条件不完全具备。

### 核心判断

不管 AI 生产工具怎么发展，**质检和入库是绕不开的环节**。AI 生成的资产越多，质检需求越大。平台的核心价值不在于"替代美术"，而在于**给所有 AI 生成的资产把好质量关**。

这是确定性最高的切入点，不依赖外部 AI 工具的成熟度。
