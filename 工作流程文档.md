# TAgent 工作流程文档

> 理清从资产制作到入库的完整链路，以及多用户协作的流程设计

---

## 一、单人工作流（本地模式）

### 1.1 完整流程

```
美术做资产（Maya/Blender/3ds Max）
    ↓
保存到本机目录（如 D:\Assets\Characters\）
    ↓
打开 TAgent（双击 exe 或 python agent.py）
    ↓
"分析 D:\Assets\Characters"
    ↓
Agent 自动执行：
  ├── 扫描目录（os.walk）
  ├── 解析 FBX（Blender subprocess）
  ├── 检查贴图（Pillow）
  ├── 命名检查（regex）
  ├── 面数检查（数值比较）
  ├── AI 推断（LLM 分类/材质/风格）
  └── 生成缩略图
    ↓
查看分析结果
  ├── 资产库浏览（带缩略图）
  ├── 详情面板（材质→贴图映射、关联资产）
  └── 审核队列（高/低置信度）
    ↓
修正问题（Agent 辅助重命名/移动）
    ↓
审核通过
    ↓
入库（复制到目标目录 + 生成 UE5 导入清单）
    ↓
在 UE5 中运行导入脚本
    ↓
SVN commit（美术手动操作）
```

### 1.2 涉及组件

| 步骤 | 组件 | 说明 |
|------|------|------|
| 扫描 | `tools/file_info.py` | os.walk 遍历目录 |
| FBX 解析 | `tools/mesh_fbx.py` + Blender | subprocess 调用 Blender headless |
| 贴图检查 | `tools/texture.py` | Pillow 读取分辨率/格式 |
| 命名检查 | `tools/naming.py` | regex 匹配前缀规则 |
| AI 推断 | `tags/inferrer.py` | LLM 分类/材质/风格 |
| 审核 | `tools/review.py` | SQLite 查询 + 置信度分组 |
| 入库 | `tools/intake.py` | 重命名 + 移动 + 生成清单 |

## 二、团队工作流（中心模式）

### 2.1 部署架构

```
美术 A 的电脑                    中心服务器
├── TAgent（本地）                ├── TAgent（中心）
├── SVN 工作副本                  ├── 共享数据库
├── Blender                      ├── 规范文档
├── 资产文件                      └── Web 前端
└── UE 引擎
```

### 2.2 美术端流程

```
1. 双击 TAgent.exe 启动本地服务
2. 浏览器自动打开 http://localhost:8080
3. "分析 D:\Project\ArtResources\Characters"
4. 查看结果，修正问题
5. 审核通过
6. 入库（本地操作）
7. Agent 自动同步元数据到中心服务器
8. SVN commit（手动）
```

### 2.3 主管端流程

```
1. 打开 http://中心服务器:8080
2. 查看项目总览（所有人的资产统计）
3. 审核待审资产（来自所有美术的分析结果）
4. 通过/拒绝/修改
5. 美术本机 Agent 收到审核结果
```

### 2.4 数据同步

```
本机 Agent                          中心服务器
    │                                  │
    ├─ 分析完成 ──POST /api/sync──────►│ 存入共享数据库
    │                                  │
    ├─ 用户修正 ──POST /api/corrections►│ 存入修正日志
    │                                  │
    │◄─ 拉取规范 ──GET /api/conventions─┤
    │◄─ 拉取规则 ──GET /api/rules──────┤
    │                                  │
    │◄─ 审核结果 ──GET /api/reviews────┤
```

## 三、SVN 集成

### 3.1 当前 SVN 工作流（无 Agent）

```
美术做资产 → 手动入端 UE → 手动配置 → SVN commit
```

### 3.2 加入 Agent 后

```
美术做资产 → Agent 分析 → 修正问题 → 入端 UE → SVN commit
                        ↓
                 同步到中心服务器
                        ↓
                 主管审核（可选）
```

### 3.3 SVN 双目录方案

```
SVN 仓库
├── ArtResources/                    ← 原始资产（FBX、贴图）
│   └── Characters/Hero/
│       ├── SK_Hero.fbx
│       └── T_Hero_D.tga
│
└── Content/                         ← UE 导入资产
    └── Characters/Hero/
        └── SK_Hero.uasset
```

- 两个目录子路径一一对应
- 入库时 Agent 复制原始文件到 ArtResources/
- UE 导入后生成 .uasset 到 Content/
- SVN commit 两个目录一起提交

### 3.4 Agent 与 SVN 的关系

| 功能 | SVN | Agent | 关系 |
|------|-----|-------|------|
| 文件版本管理 | ✅ 核心 | ❌ 不做 | 互补 |
| 资产质量检查 | ❌ | ✅ 核心 | Agent 独有 |
| 命名规范检查 | ❌ | ✅ 核心 | Agent 独有 |
| 资产审核 | ✅ 提交前 review | ✅ 分析后 review | 重叠 |
| 入库自动化 | ❌ | ✅ 核心 | Agent 独有 |

**定位**：Agent 是 SVN 提交前的"质量门禁"，不替代 SVN。

## 四、LLM 调用策略

### 4.1 调用分布

```
一次完整分析（100 FBX + 50 贴图）：

本地执行（不调 LLM）：
  ├── 扫描目录        → os.walk
  ├── 解析 FBX        → Blender subprocess（~50 分钟）
  ├── 检查贴图        → Pillow
  ├── 命名检查        → regex
  └── 面数检查        → 数值比较

LLM 调用：
  ├── 理解用户意图    → 1 次
  ├── 选择工具        → 1 次
  ├── AI 推断 50 资产  → 5 批 × 1 次 = 5 次
  ├── 生成报告        → 1 次
  └── 审核/入库交互   → 2-3 次

总计：~10 次 LLM 调用（~6 分钟）
      Blender 占 89% 耗时，LLM 占 9%
```

### 4.2 双模式 LLM

```
云端 API（GLM-5/DeepSeek）：
  ├── 优点：质量好、无需硬件
  ├── 缺点：按量付费、需要联网
  └── 适合：复杂推断、报告生成

自建模型（Qwen-14B 等）：
  ├── 优点：无费用、低延迟
  ├── 缺点：需要 GPU、质量略低
  └── 适合：简单对话、分类
```

切换方式：`/llm switch deepseek` 或前端设置页切换。

## 五、知识沉淀

### 5.1 记忆层级

| 层级 | 内容 | 存储位置 | 更新方式 |
|------|------|---------|---------|
| L0 项目画像 | 资产分布、命名规律、面数预算 | 本机 + 中心 | 自动统计 |
| L1 推断规则 | 从修正中提炼的规则 | 本机 + 中心 | 累积 10 条自动压缩 |
| L2 修正记录 | 用户纠正的原始数据 | 本机 + 中心 | 每次纠正记录 |
| L3 对话历史 | 个人对话记录 | 仅本机 | 不同步 |

### 5.2 规则来源

```
用户说 "@开头的是动画文件"
    ↓
Agent 调用 add_custom_rule(pattern="@*.*", type="animation")
    ↓
写入项目配置文件（.ta_agent/configs/project/*.yaml）
    ↓
下次分析自动生效（优先级最高）
```

### 5.3 知识流向

```
每天：美术修正 → 记录到本机 → 同步到中心
每周：中心聚合所有修正 → 提炼规则 → 更新项目画像
每次启动：本机从中心拉取最新规则和画像
```

## 七、SVN 集成策略

### 7.1 问题：美术绕过 Agent 直接提交

Agent 的质量检查只有在美术使用时才有效。如果美术直接 `svn commit`，Agent 形同虚设。

### 7.2 三种策略

**策略 1：pre-commit hook（强制）**
```
svn commit → hook 检查 → 未通过 Agent 审核 → 拒绝提交
```
- 优点：不可能绕过
- 缺点：需要配置 SVN 服务器，美术可能抵触

**策略 2：post-commit 监控（被动）**
```
svn commit → hook 触发 → Agent 自动分析 → 结果推送到中心
```
- 优点：不阻碍工作流
- 缺点：事后发现问题，需要返工

**策略 3：混合模式（推荐）**
```
ArtResources/ 提交 → 自动分析，不阻碍（日常）
Content/ 提交 → 必须审核通过才能提交（关键节点）
```

### 7.3 推荐做法

**先用被动监控，不强制**：
1. Agent 定期扫描 SVN 工作副本
2. 发现新文件或变更文件 → 自动分析
3. 结果推送到中心服务器
4. 主管审核，有问题再沟通

等团队习惯后再加 pre-commit hook。

## 八、团队推广策略

### 8.1 核心原则

**帮美术省事，不是给美术添堵。**

| 错误做法 | 正确做法 |
|---------|---------|
| "你的命名不规范" → 像挑刺 | "我帮你一键重命名 20 个文件" → 省事 |
| "必须通过审核才能提交" → 强制 | "你先提交，有问题我告诉你" → 无感 |
| "配置 config.py 和 API Key" → 麻烦 | "双击图标，拖文件夹就行" → 零门槛 |
| "分析 100 个文件等 30 分钟" → 等太久 | "先看前 5 个结果，后台跑其余的" → 即时价值 |

### 8.2 推广路径

```
第 1 周：TA 自用
  └── TA 自己分析自己的资产，熟悉工具，发现价值

第 2 周：TA 帮美术用
  └── "你把文件夹发给我，我帮你检查"
  └── 美术看到效果：面数超标、命名问题、缺失材质

第 3 周：美术自己用
  └── "你自己拖文件夹进去，比发给我快"
  └── 美术开始主动用：一键重命名、批量导入

第 4 娑：变成习惯
  └── "做完资产顺便跑一下 Agent"
  └── 不需要强制，自然形成流程
```

### 8.3 最容易被接受的功能

| 功能 | 美术痛点 | 美术感受 |
|------|---------|---------|
| 一键重命名 | 手动改 20 个文件名 | "太省事了" |
| 自动检查面数 | 不知道有没有超标 | "不用自己数了" |
| 批量导入 UE | 手动导入 50 个文件 | "一键搞定" |
| 缩略图预览 | 不知道文件长什么样 | "很方便" |
| 贴图格式检查 | 不确定格式对不对 | "帮大忙了" |

### 8.4 功能优先级（按美术接受度排序）

```
高接受度（先做）：
  ✅ 一键重命名
  ✅ 批量导入 UE
  ✅ 缩略图预览
  ✅ 贴图格式检查

中接受度（后做）：
  ⚠️ 面数/命名规范检查
  ⚠️ AI 分类推断
  ⚠️ 审核工作流

低接受度（慎做）：
  ❌ 强制 pre-commit hook
  ❌ 必须通过审核才能提交
```

### 8.5 关键指标

推广是否成功，看这些指标：
- **主动使用率**：有多少美术不用催就自己用
- **问题发现率**：Agent 发现的问题中，有多少是美术不知道的
- **修复率**：发现问题后，美术主动修复的比例
- **重复使用率**：用过一次后，下次还用的比例
