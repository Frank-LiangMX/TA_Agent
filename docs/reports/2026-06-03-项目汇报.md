## 一、TA Agent 是什么

专为游戏技术美术（TA）设计的 AI 智能体，解决美术资产从"生产 → 质检 → 入库"流程中的重复劳动问题。

**核心能力：**
- AI 自动质检：解析 FBX/贴图，校验命名、面数、分辨率等技术指标
- 智能审核：高置信度自动通过，低置信度人工确认
- 一键入库：重命名 + 移动 + 生成 UE5 导入脚本
- 语义搜索：自然语言查找资产（如"中世纪金属武器"）
- 项目记忆：纠正一次永远记住，跨会话保持

### 核心设计：三层记忆系统

Agent 的 AI 推断层本身是无状态的，用户纠正过的错误下次还会犯。记忆系统让 Agent 具备**跨会话学习能力**。

```
┌─────────────────────────────────────────────────────┐
│  L0 项目画像      长期稳定的项目规则（经多次验证）     │
│  ─────────────────────────────────────────────────── │
│  L1 纠正规则      用户纠正提炼的精简规则              │
│  ─────────────────────────────────────────────────── │
│  L2 归档          原始纠正记录，等待累积合并           │
└─────────────────────────────────────────────────────┘
```

**设计原则：**
- **无纠正不记忆**：只有用户显式纠正才写入，LLM 自己的推测不存
- **按需注入**：根据当前资产特征匹配相关规则，不全量塞入 prompt
- **自压缩**：超过阈值自动合并，token 消耗始终可控

**生命周期：** 用户纠正 → L2 归档 → 累积后 LLM 合并 → L1 规则 → 多次命中提升置信度 → 可靠规则提炼为 L0 画像

**举个例子：**

```
第 1 次：Agent 检查 "SM_Crate.fbx"，面数 8000，判定"通过"
         用户纠正："道具类上限是 5000，不是 8000"
         → 写入 L2：{ 类型: 道具, 上限: 5000, 来源: 用户纠正 }

第 2 次：Agent 检查另一个道具 "SM_Barrel.fbx"，面数 6000
         自动命中 L2 规则 → 判定"不通过"
         用户确认正确 → 规则置信度上升

第 N 次：规则被多次命中且未再被纠正
         → 提炼为 L0 项目画像："本项目道具面数上限 5000"
         → 后续所有会话自动生效
```

核心价值：**同一个错误只犯一次**，Agent 越用越懂你的项目规范。

## 三、近两周重点工作（05.20 - 06.03）

### 1. 项目架构重构 → Monorepo
- 单体项目重组为 `apps/`（应用层）+ `packages/`（共享包）+ `backend/`（核心）
- 根目录只保留薄壳入口，代码职责清晰分离
- 构建产物统一到 `release/`，文档同步整理

### 2. Electron 桌面应用
- 框架搭建 → 开发模式 → 打包流程，全流程打通
- 修复打包资源遗漏（wechat/、assets/）、窗口闪退等问题
- 打包脚本精简为 4 步，自动清理中间产物

### 3. 前端 v0.27 大版本
- 双卡片悬浮布局 + frameless 窗口适配
- 新功能：LLM 思考模式、附件上传、模型切换、用量日志
- 主题图标系统（多预设切换）
- 工作区系统 + 会话按模式隔离

### 4. 联机模式 + 中心服务器
- 搭建中心服务器框架（`apps/server/`）
- 客户端双模式：本地运行 / 联机协作
- 管理后台 + WebSocket 断开保护
- 动态端口分配 + 运行时健康检查

### 5. 发版自动化
- GitHub Actions：push tag → 自动构建 → 发布 Release
- `release.py` CLI：一条命令完成版本号同步 + commit + tag + push
- VERSION 文件作为单一版本源，避免手动改错

### 解决的关键问题

| 问题 | 影响 | 解决方案 |
|------|------|----------|
| Electron 打包后资源丢失 | 用户拿到的安装包缺 wechat/、assets/，功能不完整 | 修正打包配置，补全资源目录 |
| 重开 Electron 累积重复会话标签 | 每次重启多一个空白标签，体验差 | 修复会话初始化逻辑 |
| Monorepo 重构后 sys.path 断裂 | 薄壳入口找不到 backend/ 和 packages/ 的模块 | 统一 sys.path 修正 + 薄壳改为 `from agent_main import *` |
| 后端端口冲突 | 本地已占用 8080 时启动失败 | 动态端口分配 + 启动前健康检查 |
| WebSocket 断开后继续发消息报错 | 前端断连后后端抛异常 | send_event/send_response/send_error 加断开保护 |
| 不同模式会话互相干扰 | 本地/联机模式共用同一份会话数据 | 按 agent mode 隔离 workspace sessions |
| CI 发版产物缺失 | Release 页面没有 latest.yml 和 blockmap，自动更新失效 | 修正 workflow 上传逻辑 |
