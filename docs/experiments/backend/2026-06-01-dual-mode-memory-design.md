# 双模式记忆系统设计框架（TA 模式 + 通用 Agent 模式）

> 创建日期：2026-06-01  
> 状态：设计中（待评审）  
> **实现进度**：见 [工作台双模式实施台账](./2026-06-01-workbench-dual-mode-roadmap.md)（与本设计稿 Phase 对齐）

---

## 1. 背景与目标

当前系统定位为游戏资产管理垂直 Agent（TA 模式），记忆体系围绕资产推断与纠正构建。随着新增通用 Agent 模式（办公协作 + vibe coding），两类任务语义差异显著，若继续共享单一记忆空间，会出现串味、误召回与规则污染。

本设计目标：

1. 支持 TA 模式与通用模式的记忆隔离；
2. 保留可控的跨模式用户偏好复用能力；
3. 修复现有 L0 项目画像缺少自动触发更新的问题；
4. 以最小改动优先落地，避免一次性大重构。

---

## 2. 现状问题

### 2.1 结构层面

- 现有记忆为单命名空间（L0/L1/L2），无法区分模式来源；
- TA 规则与通用办公/编码偏好被混存时，召回相关性不可控。

### 2.2 触发层面

- L0 画像依赖 `update_project_profile` 工具调用，缺少框架级自动流程；
- 记忆写入行为对模型自主性依赖较高，稳定性不足。

### 2.3 运维层面

- 缺少模式维度统计（无法判断某模式是否“学到东西”）；
- 切换模式后记忆可见性边界不清晰，难以向用户解释。

---

## 3. 设计原则

1. **默认隔离**：不同模式记忆不自动互通；
2. **显式共享**：跨模式同步只能通过明确策略或用户操作触发；
3. **自动维护**：L0/L1/L2 的演进由 pipeline 触发，不依赖 prompt 提示；
4. **低侵入改造**：优先复用现有 `MemoryProvider` 接口与统计逻辑；
5. **可审计**：每条记忆带上模式、来源与时间元信息。

---

## 4. 目标架构（双模式 + 全局偏好）

### 4.1 记忆分区

- `global_user_profile`：跨模式共享的用户偏好（语言、风格、表达习惯等）；
- `ta`：TA 模式专属 L0/L1/L2；
- `general`：通用模式专属 L0/L1/L2。

### 4.2 目录建议

```text
runtime/memory/
  global/
    profile.json
  ta/
    profile.md
    rules.json
    corrections.jsonl
  general/
    profile.md
    rules.json
    corrections.jsonl
```

### 4.3 注入策略

- TA 模式：注入 `global_user_profile + ta.L0 + ta.L1(按相关性)`；
- 通用模式：注入 `global_user_profile + general.L0 + general.L1(按相关性)`；
- 不允许直接注入另一模式的 L1/L2。

---

## 5. TA 模式 L0 处理策略

TA-L0 定义为“项目资产稳定约束摘要”，仅保留跨会话稳定规则：

- 项目风格与资产方向；
- 命名规范与目录约定；
- 质量阈值（面数/贴图/材质等）；
- 已验证的项目例外规则。

不进入 TA-L0 的内容：

- 单次会话结论；
- 单资产临时纠正；
- 一次性上下文信息。

### 5.1 自动更新触发

在以下事件自动触发 L0 提炼任务：

1. L2 -> L1 压缩完成后；
2. L1 规则数量或置信度变化达到阈值时；
3. 会话结束时（若本会话有有效纠正）。

### 5.2 提炼门槛

- 规则命中次数达到最低阈值；
- 最近窗口内无反向纠正；
- 置信度高于最小门槛；
- 去重归并后总长度受控（建议 <= 500 tokens）。

---

## 6. 通用模式记忆策略

通用模式重点不在资产规则，而在“用户偏好 + 工作流习惯 + 项目常识”：

- 建议将“用户表达偏好”优先写入 `global_user_profile`；
- 将“项目技术约定/命令习惯”写入 `general` 模式记忆；
- 避免将短期任务进度写入长期记忆（此类应留在会话系统）。

---

## 7. 接口与实现改造点（最小改动）

### 7.1 Provider 层

- 为 `FileMemoryProvider` 增加 `namespace/mode` 参数；
- 内部读写路径切换到 `runtime/memory/{namespace}`；
- 保持现有 `get_memory_stats` 返回结构，新增 `namespace` 字段。

### 7.2 运行时层

- 增加“当前模式”上下文来源（会话级/配置级）；
- 初始化 provider 时按模式绑定命名空间；
- 在 post-turn 增加记忆 pipeline 调度（压缩 + L0 提炼）。

### 7.3 前端层

- 设置页显示“当前模式记忆统计”；
- 明确展示当前模式来源（TA / General）；
- 预留“同步到全局偏好”入口（后续阶段）。

## 8. 分阶段实施计划

| Phase | 目标 | 预计工作量 | 风险 |
|-------|------|------------|------|
| P1 | 模式隔离存储 + 按模式注入 + 统计分区 | 2-4 天 | 低 |
| P2 | 自动 L0 更新 pipeline（TA/General） | 1-2 天 | 中 |
| P3 | 全局偏好共享与手动同步策略 + 通用模式只读 subagent 并行 | 3-5 天 | 中 |
| P4 | 可视化审计与迁移工具 + 协作编排治理 | 3-5 天 | 中 |

## 9. 多 Agent 协作设计（增量）

### 9.1 目标

在通用模式提升并行吞吐（搜索、编码、文档整理等），同时保证 TA 模式核心链路稳定。

### 9.2 范围策略

- 通用模式：支持并行 subagent（优先只读任务）；
- TA 模式：首阶段关闭 subagent 写操作，仅允许受控只读分析；
- TA 的审核、入库、状态写回保持主线程串行。

### 9.3 记忆边界

- subagent 默认不直接写长期记忆；
- subagent 输出先回主 agent 汇总；
- 仅主 agent 通过统一 pipeline 判定是否写入记忆。

### 9.4 上线开关

- `subagent.enabled.general = true`
- `subagent.enabled.ta = false`（首版默认）
- 灰度开启后再评估 TA 场景的可并行任务白名单。

---

## 10. 验收标准

1. 模式切换后，召回内容不跨模式污染；
2. TA 模式在连续纠正后可自动生成/更新 L0；
3. 通用模式可独立积累办公/编码相关记忆；
4. 记忆统计可按模式独立观测；
5. 通用模式可并行执行只读 subagent 任务，结果可回收合并；
6. TA 核心流程（分析/审核/入库）行为与现网保持一致；
7. 不破坏现有单模式数据（提供兼容迁移策略）。
