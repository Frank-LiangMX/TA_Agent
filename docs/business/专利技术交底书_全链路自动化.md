# 技术交底书

## 发明名称

一种基于 AI Agent 的游戏资产全链路自动化管理方法及系统

## 技术领域

本发明涉及人工智能与游戏开发技术领域，特别涉及一种利用大语言模型自主决策能力，实现游戏资产从质检、审核到入库的全流程自动化管理方法。

---

## 一、背景技术

### 1.1 现有技术的不足

游戏开发中，美术资产（3D 模型、贴图、材质等）在入库引擎前需要经过严格的质检流程。当前业界存在以下问题：

1. **人工质检效率低**：技术美术（TA）需要逐个检查资产的命名规范、面数预算、贴图格式等，一个资产需要 5-15 分钟，批量处理耗时巨大。

2. **纯规则脚本缺乏语义理解**：现有的自动化检查工具（如 UE5 Asset Validation）基于硬编码规则，无法理解资产的语义信息（如分类、材质、风格），只能做格式层面的校验。

3. **流程碎片化**：质检、审核、入库是三个独立环节，使用不同工具，需要人工串联，无法实现端到端自动化。

4. **现有 AI 工具覆盖不全**：
   - AI 3D 生成工具（Meshy、Tripo）只管生成，不管质检和入库
   - 生产管理工具（ShotGrid）只管进度，不管资产内容
   - 通用 AI Agent（AutoGPT、LangChain Agent）缺乏游戏资产领域的专业知识和工具集

### 1.2 本发明要解决的技术问题

如何利用 AI Agent 的自主决策能力，将游戏资产的质检、审核、入库三个环节自动串联，实现端到端的全流程自动化，同时保证质检的语义级准确性。

## 二、发明内容

### 2.1 技术方案概述

本发明提出一种基于大语言模型（LLM）的游戏资产全链路自动化管理方法，核心创新在于：

1. **领域专用工具集**：为 AI Agent 配备游戏资产领域的专用工具（FBX 解析、贴图检查、命名校验、AI 视觉推断等），通过 Function Calling 机制供 LLM 自主调用。

2. **自主工作流编排**：LLM 基于 ReAct（Reasoning + Acting）模式，根据任务目标自主决定调用哪些工具、以什么顺序执行、何时需要人工介入。

3. **置信度分级审核机制**：AI 推断结果附带置信度评分，高置信度项自动通过，低置信度项提交人工审核，在效率和准确性之间实现最优平衡。

4. **端到端入库执行**：审核通过后自动执行重命名、目录创建、文件移动、引擎导入配置生成，完成从"原始资产"到"引擎可用"的全链路。

### 2.2 详细技术方案

#### 2.2.1 系统架构

```
用户（自然语言指令）
    ↓
AI Agent（LLM + ReAct 循环）
    ├── 工具调度层：根据 LLM 决策调用对应工具
    ├── 领域工具集：
    │   ├── 资产扫描工具：遍历目录，识别 FBX/贴图/材质文件
    │   ├── FBX 解析工具：提取面数、顶点数、骨骼、UV、包围盒
    │   ├── 贴图检查工具：检查分辨率、格式、通道数、色彩空间
    │   ├── 命名校验工具：检查文件名是否符合项目命名规范
    │   ├── AI 推断工具：多模态 LLM 分析资产分类、材质、风格
    │   ├── 审核管理工具：分级审核、批量通过、审核提交
    │   └── 入库执行工具：重命名、移动、生成导入配置
    └── 状态管理层：资产状态机（pending → approved → imported）
```

#### 2.2.2 自主工作流编排方法

AI Agent 的工作流编排不是预定义的固定流程，而是 LLM 基于当前上下文自主决策的动态流程：

**步骤 1：任务理解**
LLM 解析用户的自然语言指令，理解任务目标（如"分析某目录的资产并入库"）。

**步骤 2：规划执行**
LLM 根据任务目标，决定需要调用的工具和执行顺序。例如：
- 先调用项目配置检查工具，确认是否有项目规范
- 再调用资产扫描工具，获取资产列表
- 逐个调用 FBX 解析、贴图检查、命名校验工具
- 调用 AI 推断工具进行语义级分类

**步骤 3：工具调用与结果观察**
Agent 执行工具调用，将结果返回 LLM。LLM 观察结果后决定下一步行动：
- 如果发现异常，记录并继续处理其他资产
- 如果需要人工确认，暂停并询问用户
- 如果所有资产处理完毕，进入审核阶段

**步骤 4：迭代执行**
重复步骤 2-3，直到任务完成或达到最大迭代次数。

#### 2.2.3 置信度分级审核方法

AI 推断工具在分析资产时，为每个推断字段（分类、材质、风格等）返回置信度评分（0-100%）。基于置信度执行分级审核策略：

| 置信度区间 | 处理策略 |
|-----------|---------|
| ≥ 90%（高置信度） | 自动通过，记录推断结果 |
| 70%-89%（中置信度） | 批量展示，用户可一键全部通过 |
| < 70%（低置信度） | 逐个展示详情，用户必须逐项确认 |

用户对低置信度项的确认/修改结果会被记录，用于后续推断的参考。

#### 2.2.4 端到端入库执行方法

审核通过后，自动执行以下入库步骤：

**步骤 1：名称规范化**
根据项目配置的命名规则，生成规范名称。例如：
- 原始名称：`sword_01.fbx`
- 规范名称：`SM_Weapon_Sword_01.fbx`

**步骤 2：名称合规校验**
对生成的规范名称进行合规性检查，确保符合命名规范。

**步骤 3：目录结构创建**
根据资产分类，创建对应的引擎目录结构。例如：
- 武器类：`/Game/Weapons/`
- 建筑类：`/Game/Environment/Buildings/`

**步骤 4：文件移动**
将 FBX 文件和关联贴图文件移动到目标目录，贴图自动重命名以匹配新的资产名称。

**步骤 5：导入配置生成**
生成引擎导入配置文件（JSON 格式），包含：
- 资产路径和类型
- 导入参数（缩放、LOD、碰撞、材质导入等）
- 元数据（分类、材质、风格等）

**步骤 6：导入脚本生成**
生成可直接在引擎中执行的 Python 导入脚本，读取配置文件并调用引擎 API 完成批量导入。

**步骤 7：状态更新**
更新资产状态为"已入库"，记录入库时间和路径。

#### 2.2.5 工具调用异常处理

在自主工作流执行过程中，对各类异常进行容错处理：

| 异常类型 | 处理策略 |
|---------|---------|
| LLM API 超时/限流 | 自动重试（指数退避），超过重试次数后降级为规则模式 |
| 资产文件损坏 | 跳过该资产，记录错误日志，继续处理其他资产 |
| 工具执行失败 | 返回错误信息给 LLM，由 LLM 决定是否重试或跳过 |
| JSON 解析失败 | 自动修复常见问题（如 Windows 路径反斜杠转义），重试解析 |

## 三、与现有技术的区别

| 对比维度 | 现有技术 | 本发明 |
|---------|---------|--------|
| 质检方式 | 纯规则脚本（UE5 Validation） | AI Agent + 规则双重检查 |
| 语义理解 | 无（只能检查格式） | AI 推断分类、材质、风格 |
| 流程编排 | 预定义固定流程 | LLM 自主决策动态编排 |
| 审核机制 | 无（通过/不通过二元判断） | 置信度分级审核 |
| 端到端能力 | 需要多个工具人工串联 | 一个 Agent 完成全流程 |
| 异常处理 | 简单报错 | 容错降级，继续执行 |

**关键区别**：现有技术中，游戏资产的质检、审核、入库是三个独立环节，由不同工具完成，需要人工串联。本发明通过 AI Agent 的自主决策能力，将三个环节自动串联为一个端到端流程，且通过置信度机制在自动化和准确性之间实现最优平衡。

## 四、有益效果

1. **效率提升**：20 个资产的处理时间从 2.5 小时缩短到 7 分钟（20 倍以上）。

2. **语义级质检**：不仅检查格式规范，还能理解资产的分类、材质、风格等语义信息。

3. **自适应工作流**：AI Agent 根据实际资产情况动态调整处理策略，而非执行固定脚本。

4. **分级审核**：高置信度自动通过，低置信度人工确认，兼顾效率和准确性。

5. **端到端自动化**：从原始资产到引擎可用，全流程无需人工干预（低置信度项除外）。

## 六、具体实施方式

### 6.1 实施例一：批量资产质检与入库

用户输入：`"帮我分析 D:\Assets\NewBatch 并入库到 D:\UE5\Content"`

AI Agent 自主执行：
1. 调用项目配置检查工具 → 发现无配置 → 询问用户是否创建
2. 调用资产扫描工具 → 发现 15 个 FBX + 42 张贴图
3. 逐个调用 FBX 解析工具 → 提取面数、骨骼等几何信息
4. 调用贴图批量检查工具 → 检查分辨率、格式
5. 调用命名校验工具 → 发现 3 个文件命名不规范
6. 调用 AI 推断工具 → 推断每个资产的分类、材质、风格
7. 输出分析报告，询问用户是否进入审核
8. 调用审核工具 → 高置信度批量通过，低置信度展示详情
9. 用户确认后，调用入库工具 → 重命名、移动、生成导入配置

### 6.2 实施例二：异常容错

在处理过程中，第 5 个 FBX 文件损坏导致解析失败：
- 系统记录错误日志，跳过该文件
- 继续处理剩余 14 个资产
- 最终报告：14 个成功，1 个失败（附失败原因）

### 发明专利 vs 实用新型

| 维度 | 发明专利 | 实用新型 |

| 保护对象 | 方法、工艺、系统 | 产品的形状/结构 |
| 适用软件 | ✅ 方法专利可以保护软件算法和流程 | ❌ 不保护方法和算法 |
| 审查周期 | 18-36 个月 | 6-12 个月 |

| 审查标准 | 新颖性 + 创造性 + 实用性 | 新颖性 + 创造性（低于发明） |
| 费用 | 较高 | 较低 |

**本发明应申请发明专利。**

3. 本发明的创新点在于"AI Agent 自主决策串联多工具"的方法，属于发明专利的保护范围
