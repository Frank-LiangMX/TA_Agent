# TAgent 用户认证与分布式架构设计

> 日期：2026-05-20
> 状态：待确认（需与公司技术人员确认 SSO 集成方案）

---

## 一、背景

TAgent 需要支持多用户协作场景：
- 用户本地执行 Agent（分析资产、Blender 解析）
- 服务器存储共享数据（资产数据库、审核记录、项目配置）
- 用户身份需要统一认证
- 项目配置需要统一管理

公司已有内部 SSO 系统（XSJSSO），已在 Blender 插件中集成。

## 三、登录流程

### 3.1 流程图

```
Electron 启动
    │
    ▼
检查本地凭证是否存在且有效
    │
    ├─ 有效 → 自动登录 → 进入主界面
    │
    └─ 无效/不存在
        │
        ▼
    显示登录窗口
        │
        ├─ 用户输入账号密码
        │
        ▼
    调用 XSJSSO API 验证
        │
        ├─ 成功 → 保存凭证 → 获取用户信息 → 进入主界面
        │
        └─ 失败 → 显示错误 → 重新输入
```

### 3.2 登录接口

```
POST https://intranet-xsjsso.seasungame.com/api-xsjsso/account/checklogin/v2

请求参数：
- appid: 应用ID（TAgent 专用）
- nonce: 当前时间戳
- sign: 签名（MD5）

请求体：
{
  "username": "zhangsan",
  "password": "AES加密后的密码",
  "ip": "10.11.131.124"
}

响应：
{
  "messageCode": { "code": 0 },
  "data": {
    "success": true,
    "userId": "zhangsan",
    "userName": "张三"
  }
}
```

### 3.3 用户信息获取

```
GET https://intranet-xsjsso.seasungame.com/api-xsjsso/account/info

参数：
- userId: 用户ID
- sign: 签名

响应：
{
  "data": {
    "userId": "zhangsan",
    "userName": "张三",
    "department": "技术美术组",
    "email": "zhangsan@company.com"
  }
}
```

## 四、配置文件设计

### 4.1 配置文件位置

```
Windows: %APPDATA%/tagent-desktop/config.json
Mac: ~/Library/Application Support/tagent-desktop/config.json
Linux: ~/.config/tagent-desktop/config.json
```

### 4.2 配置文件格式

```json
{
  "version": 1,

  "server": {
    "host": "10.11.131.124",
    "port": 8080
  },

  "auth": {
    "loginEndpoint": "https://intranet-xsjsso.seasungame.com",
    "appId": "tagent",
    "userId": "zhangsan",
    "userName": "张三",
    "role": "user",
    "lastLogin": "2026-05-20T10:30:00"
  },

  "local": {
    "projectRoot": "D:\\Projects\\MyGame",
    "blenderPath": "D:\\Program Files\\Blender Foundation\\Blender 4.3\\blender.exe"
  }
}
```

### 4.3 凭证存储

```
登录凭证（密码）不存储在配置文件中
只存储登录状态和用户信息

密码处理：
1. 用户输入密码
2. AES 加密后发送给 SSO
3. 验证成功后，只保存 userId/userName
4. 密码不保存，下次启动需要重新输入（或使用 token）
```

## 五、权限管理

### 5.1 用户角色

```
普通用户（美术）：
├── 查看项目配置（只读）
├── 提交资产分析
├── 查看自己的审核记录
└── 使用本地记忆系统

管理者（组长/TA）：
├── 普通用户权限
├── 修改项目配置（风格规范、命名规则等）
├── 审核资产
├── 查看所有用户的资产
└── 查看团队记忆规则

管理员：
├── 管理者权限
├── 管理用户权限
└── 系统配置
```

### 5.2 权限矩阵

| 数据/操作 | 普通用户 | 管理者 | 管理员 |
|-----------|---------|--------|--------|
| 项目配置 | 只读 | 读写 | 读写 |
| 资产数据 | 查看+提交 | 查看+审核 | 全部 |
| 审核记录 | 查看自己的 | 查看所有 | 全部 |
| 记忆规则 | 使用 | 使用+编辑 | 全部 |
| 用户权限 | - | - | 管理 |

### 5.3 API 权限设计

```
GET  /api/projects/{id}/config      # 所有角色可读
PUT  /api/projects/{id}/config      # 仅管理者可写

GET  /api/assets                    # 所有角色可读
POST /api/assets/sync               # 所有角色可写（提交自己的分析）

GET  /api/reviews                   # 所有角色可读
POST /api/reviews/{id}/approve      # 仅管理者可写

GET  /api/memory/rules              # 所有角色可读
PUT  /api/memory/rules              # 仅管理者可写

GET  /api/users                     # 仅管理员可读
PUT  /api/users/{id}/role           # 仅管理员可写
```

## 六、数据存储设计

### 6.1 存储位置

```
服务器存储（共享）：
├── 项目配置（风格规范、命名规则、目录结构）
│   └── 管理者可修改，普通用户只读
│
├── 资产数据库
│   └── 所有人可查看，管理者可审核
│
├── 审核记录
│   └── 所有人可查看
│
├── 记忆规则（L1）
│   └── 团队共享，管理者可编辑
│
└── 用户权限
    └── 管理员管理

客户端存储（本地）：
├── 会话记录（私有）
├── 纠正记录（L2，私有）
├── 本地缓存（项目配置、规则）
└── 登录凭证（临时）
```

### 6.2 数据库设计

```
SQLite（当前）：
├── assets 表（资产数据）
├── reviews 表（审核记录）
├── projects 表（项目配置）
├── rules 表（记忆规则）
└── users 表（用户权限）

PostgreSQL（未来）：
└── 当数据量超过 10GB 或需要多项目管理时迁移
```

### 6.3 预览图存储

```
预览图不存数据库，存文件系统

数据库：
└── 只存路径引用
    {
      "asset_id": "xxx",
      "preview_thumbnail": "/previews/xxx_thumb.png",
      "preview_front": "/previews/xxx_front.png"
    }

文件系统：
└── /data/previews/
    ├── asset_001_thumb.png
    ├── asset_001_front.png
    └── ...
```

## 七、记忆系统多用户设计

### 7.1 三层结构

```
L0 项目画像：
├── 存储位置：服务器
├── 大小限制：500 tokens（约 2000 字符）
├── 更新方式：管理者手动 + Agent 自动学习
└── 权限：管理者可编辑，普通用户只读

L1 规则：
├── 存储位置：服务器
├── 大小限制：15 条
├── 更新方式：自动从纠正记录压缩
└── 权限：管理者可编辑，普通用户只读

L2 纠正记录：
├── 存储位置：本地 + 服务器（可选）
├── 大小限制：10 条触发压缩
├── 更新方式：用户纠正时记录
└── 权限：用户提交自己的，管理者查看所有
```

### 7.2 数据量控制

```
单个项目：
├── L0 项目画像：~2KB（恒定）
├── L1 规则：15 条 × 500 字节 = ~7.5KB（恒定）
└── L2 纠正记录：10 条 × 1KB = ~10KB（压缩后清空）

总计：~20KB/项目（恒定，不会无限增长）
```

### 7.3 压缩机制

```
用户纠正 10 次
    ↓
L2 纠正记录达到阈值
    ↓
自动压缩为规则
    ↓
删除原始记录，保留规则
    ↓
规则超过 15 条
    ↓
淘汰低置信度规则
```

### 7.4 学习流程

```
用户 A 纠正资产：
  "SM_Chair 不是 building/furniture，而是 prop/furniture"
    ↓
  Agent 记录到本地 L2
    ↓
  同步到服务器 L2（可选）
    ↓
  服务器自动压缩为规则
    ↓
  规则同步到所有客户端

管理者手动调整：
  管理者编辑项目画像
    ↓
  存服务器
    ↓
  所有客户端同步
```

### 7.5 优先级

```
Agent 推断时：
1. 服务器项目画像（L0）→ 高优先级
2. 服务器规则（L1）→ 高优先级
3. 本地纠正记录（L2）→ 低优先级
4. 默认规范 → 兜底
```

## 八、角色与职责

### 8.1 服务器（TAgent Server）

```
职责：
├── 存储资产数据库（团队共享）
├── 存储审核记录（团队共享）
├── 存储项目配置（团队统一）
├── 存储记忆规则（团队共享）
├── 提供 Web UI（查看统计）
└── 验证用户身份（调用 XSJSSO）

不负责：
├── 不执行 Agent
├── 不执行 Blender
└── 不存储用户密码
```

### 8.2 客户端（Electron）

```
职责：
├── 用户登录（调用 XSJSSO）
├── 本地执行 Agent
├── 本地执行 Blender
├── 本地资产文件
└── 同步数据到服务器

存储：
├── 本地配置文件
├── 会话记录（本地）
├── 纠正记录（本地）
├── 本地缓存（项目配置、规则）
└── 登录凭证（临时）
```

## 九、数据流

### 9.1 登录流程

```
用户 → Electron → XSJSSO 服务器
         │
         ├─ 验证成功
         │   ├─ 保存用户信息到本地配置
         │   └─ 连接 TAgent 服务器
         │
         └─ 验证失败
             └─ 显示错误信息
```

### 9.2 配置同步流程

```
客户端启动：
1. 登录（SSO）
2. 获取用户角色
3. 拉取项目配置（只读或可编辑）
4. 拉取记忆规则
5. 本地缓存

管理者修改配置：
1. 修改项目配置
2. 提交到服务器
3. 服务器广播通知其他客户端
4. 其他客户端重新拉取配置
```

### 9.3 业务流程

```
用户操作 → Agent 本地执行 → 分析结果
                              │
                              ├─ 存储到本地会话
                              └─ 同步到 TAgent 服务器
                                    │
                                    ├─ 资产数据库更新
                                    └─ 审核记录更新
```

## 十、安全考虑

| 项目 | 措施 |
|------|------|
| 密码传输 | AES 加密 |
| 密码存储 | 不存储，每次输入 |
| 接口签名 | MD5 签名验证 |
| 会话安全 | 使用 token（如果 SSO 支持） |
| 数据传输 | HTTPS（内网可考虑 HTTP） |
| 权限控制 | 基于角色的访问控制 |
